# Guest Checkout Phase 1 Order Number Contract

## Purpose

This document locks the canonical order number contract for guest checkout Phase 1.
It defines exactly where `order_number` comes from and what other layers are not allowed to do.

## Canonical Source

- Backend service is the only canonical source of `order_number`.
- `order_number` must be generated by backend application logic.
- Frontend must never fabricate, derive, or guess `order_number`.
- Database must not generate a competing `order_number` value.

## Database Rule

- Database keeps uniqueness enforcement for `orders.order_number`.
- Database does not own order number generation.
- Any DB-generated `order_number` behavior must be removed or disabled.

This includes trigger-based or function-based generation logic.

## Frontend Rule

- Frontend must only display the `order_number` returned by backend responses.
- Frontend must not build an order number from:
- internal numeric `id`
- timestamps
- local formatting rules
- mock placeholders

## Cross-System Consistency Rule

The same backend-generated `order_number` must be reused consistently across:

- Order creation response
- Guest success page
- Confirmation email
- Guest lookup flow
- Order detail response

No layer is allowed to reformat or replace the canonical value with another numbering scheme.

## Existing Drift To Eliminate

The current system contains order number drift that must be treated as invalid for the target design:

- Backend service-generated order number behavior
- Database trigger/function-generated order number behavior
- Frontend-generated or frontend-formatted order number display

Phase 1 schema and migration work must remove this drift.

## Phase 1 Required Outcome

After Phase 1 schema alignment is complete:

- backend service remains the only source of `order_number`
- database only enforces uniqueness
- fresh database boot and migrated database boot must behave the same way

## Non-Goals

This document does not define:

- the final visual format of `order_number`
- lookup token design
- post-order security hardening

It only locks ownership of order number generation and consistency responsibilities.
